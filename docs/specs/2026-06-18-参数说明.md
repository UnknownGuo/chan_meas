# 处理流程关键参数说明

目的：把目前锁死在代码里、UI 摸不到的那些参数列出来，说清楚每个参数"调大/调小会发生什么"，方便你判断哪些值得做成可调界面，哪些维持代码里写死就行。

不是 API 文档，只挑"会实际影响结果"的参数；纯工程类的参数（比如 GPU 开关、numpy/torch 选择）不在这里。

---

## 1. B2B 校准参数

代码位置：`src/calibration/b2b_frequency.py::regularized_frequency_calibrate`，常数集中在 `src/calibration/constants.py`。

### 1.1 `attenuation_db`（当前值：60.0）

**这是什么**：录 B2B 背靠背参考信号时，线缆里插了多少 dB 的衰减器（防止直连把接收机打饱和）。真实的车对车测量没有这个衰减器，所以做"除以系统响应"的解卷积之前，要先把 B2B 参考信号的幅度按这个值放大回去，否则结果会带着衰减器的损耗，整体多出虚的增益。

**调大/调小会怎样**：
- 这个值不是"调出更好效果"的旋钮，是"测量时实际插了多少衰减就填多少"——填错了，所有功率值整体平移一个固定的dB数（填60就该填60，不是越大越好）。
- 验证方法：找一段已知传播距离的近距离测量，反推自由空间路径损耗，看算出来的功率是否落在物理合理范围。如果整体偏移了一个固定常数，说明这个值填错了。

**影响范围**：所有用同一份 `b2b_cir.npy` 做校准的输出——PDP、SAGE多径功率、Doppler-Delay谱全部受影响，因为它们都是从校准后的CIR算出来的。

### 1.2 `regularization`（当前值：1e-3，相对 `max(|H_b2b|²)` 的比例）

**这是什么**：频域解卷积 `H_cal = H_meas·conj(H_b2b) / (|H_b2b|² + λ)` 里的 λ。B2B 参考信号在某些频率点上可能很弱（甚至接近0），直接除会在那些频点上产生巨大噪声/伪迹（除以接近0的数）。λ 的作用是"垫"在分母里，防止这种除零爆炸。

**调大**：解卷积更稳，但在 B2B 信号弱的频段，校准效果变弱（那些频段基本不补偿了，相当于放弃修正那部分频响）。
**调小**：解卷积更"激进"，能修正的频响范围更宽，但 B2B 弱信号频段的噪声会被放大，结果里可能出现尖刺状伪迹。

**怎么判断合适**：看校准后 CIR 在远离主径的"噪声地板"区域是否平稳；如果出现孤立的异常高功率尖刺，通常是 λ 太小。

## 2. SAGE 自适应多径估计参数

代码位置：`src/signal/sage_adaptive.py::estimate_window_paths_adaptive`。这是核心算法，但目前从 UI 到这里的参数链路只接通了极少几个（见第4节）。

### 2.1 时间窗口：`window_size_frames`(20) / `step_frames`(100)

**这是什么**：SAGE 不是逐帧算的，是把连续若干帧（慢时间维度）打成一个窗口一起做联合估计，每隔 `step_frames` 帧滑动一次窗口。

**调大 `window_size_frames`**：
- Doppler 分辨率变好（FFT点数固定时，更长的慢时间窗口=更细的频率分辨率，`Δf = frame_rate_hz / window_size_frames`）
- 时间分辨率变差——一个窗口跨越的物理时间更长，如果场景在这段时间内变化快（比如车速快、多径快速生灭），窗口内的"联合估计"假设（窗口内信道近似不变）会失效，估计出来的多径其实是窗口内变化的平均/混叠
- 计算量增加（窗口越大，每次 SAGE 迭代里的矩阵运算越大）

**调小**：反过来，时间分辨率更好，但 Doppler 分辨率变差，且单窗口样本数少，估计噪声更大（统计意义上不够稳健）。

**`step_frames` 的影响**：纯粹是"输出多密"的问题——步长小=滑动窗口重叠多，输出的多径轨迹时间点更密，但不会让单次估计质量变好（重叠窗口之间高度相关）。`step_frames < window_size_frames` 时窗口会重叠；目前 `step_frames=100 > window_size_frames=20`，意味着窗口之间是不连续的，中间frames被跳过没有参与SAGE估计（只参与了别的图，比如CIR瀑布图）。这点你可能要确认是否符合预期。

**经验法则**：车速越快/场景变化越快，`window_size_frames` 应该越小（牺牲Doppler分辨率保时间分辨率）；静态或慢速场景可以放大窗口换取更干净的Doppler估计。

### 2.2 距离门限：`delay_gate_distance_m`（当前值：2000.0）

**这是什么**：只在"单程传播距离 ≤ 这个值"对应的延迟范围内找多径，超出范围的延迟bin直接不参与估计（噪声地板也排除在外，减少无效搜索）。

**调大**：能看到更远的多径（比如远处反射体），但搜索范围变大，计算量增加，且如果数据本身没有那么远的真实多径，纯粹是在搜噪声，容易把噪声峰误判成"远处多径"。

**调小**：聚焦近距离，计算更快，但如果实际场景里有超出门限的真实反射径，会被直接漏掉（不是因为算法找不到，是代码层面直接把那部分延迟bin砍掉了）。

**怎么定**：根据场景的物理收发距离量级来定，比如车对车几十到几百米的场景，2000米的门限其实偏松；如果你有具体场景的距离先验，应该收紧这个值换计算速度和减少假阳性。

### 2.3 停止条件：`coverage_target`(0.95) / `min_coverage_gain`(0.005) / `max_paths_hard`(30)

**这是什么**：SAGE 不是固定估计N条径，是"一条一条加，加到能量覆盖率达标就停"的自适应流程。
- `coverage_target`：重建出来的多径功率总和 / 原始PDP总能量，达到这个比例就停（比如0.95=用估计出的径解释了95%的能量）。
- `min_coverage_gain`：如果新加的一条径让覆盖率提升小于这个值（比如0.5%），认为这条径没有实质贡献，撤销它并停止（防止为了凑覆盖率硬塞进很多没意义的弱径）。
- `max_paths_hard`：无论覆盖率有没有达标，硬性上限，防止极端情况下无限加径。

**调高 `coverage_target`**：会尝试拟合更多/更弱的径来逼近100%覆盖，结果里径的数量增多，但弱径的估计噪声更大（信噪比低的成分本身就不容易估准）。
**调低**：只保留最强的主要径，结果更"干净"但可能漏掉真实存在的弱径（比如远距离散射体、低能量反射径）。

**`min_coverage_gain` 调大**：更容易提前停（更保守，只留强径）。**调小**：更容易继续加径（更激进，愿意为了一点点覆盖率增加也保留弱径）。

### 2.4 局部搜索范围：`delay_search_bins`(8) / `doppler_search_half_span_hz`(3.0) / `doppler_search_points`(81)

**这是什么**：初始候选径定出大致的延迟/Doppler位置后，`_fit_atom` 会在这个位置附近做一次局部穷举搜索（延迟±8个bin，Doppler±3Hz、81个采样点），找到让残差能量最小的精确(delay, Doppler, 幅度)组合。

**调大搜索范围**：能修正初始候选定位偏差较大的情况（比如初始PDP峰值定位本身就不准），但计算量随搜索范围线性甚至更快增长，且范围太大容易"滑"到旁边一条径的位置上（如果两条径靠得近）。

**调小**：计算快，但如果初始候选定位有偏差，精修阶段修不回来，最终估计的delay/Doppler会有系统性偏差。

**`doppler_search_points`**：决定Doppler网格的细密程度，点数越多，Doppler估计的理论精度越高，但每多一个点都要重新计算一次最小二乘投影，计算量线性增加。

### 2.5 脉冲核宽度：`pulse_half_width_bins`(4)

**这是什么**：SAGE 假设每条径在延迟域上的"形状"是一个固定宽度的脉冲（默认是半宽4个bin的高斯型核），不是单个delta脉冲。这个宽度应该匹配你的发射脉冲/匹配滤波器实际的主瓣宽度（由信号带宽决定，带宽越大，脉冲越窄）。

**核宽度设置和实际信号不匹配会怎样**：
- 设置过宽：相邻的真实多径会被这个核"糊"在一起，本来该是两条径的能量被算成一条径的能量，多径数量被低估。
- 设置过窄：核本身的能量集中度和真实回波形状不符，残差拟合质量下降，可能在真实单一径周围"分裂"出多个虚假弱径（算法误以为残差里还有没拟合干净的成分）。

**这个值理论上应该跟着带宽算出来，不应该是个写死的常数**——如果以后换了不同带宽的硬件/不同的脉冲成形滤波器，这个值需要重新校准，目前没有任何地方提示这一点。

### 2.6 弱径剔除阈值（硬编码在 `sage_adaptive.py:316`，不是参数）

```python
if pruned and enable_weak_nonprominent_prune and local_prominence_db < 3.0 and rel_power_db < -15.0:
    continue  # 剔除这条径
```

**这是什么**：最终结果里，如果一条径"局部突出度"（比邻域噪声地板高多少dB）小于3dB，**并且**功率比最强径低15dB以上，就把它从最终结果里剔除（认为它更像噪声而不是真实路径）。

**两个数字写死在代码里**，不是函数参数（外层只有 `enable_weak_nonprominent_prune` 这个开关能控制要不要做这个剔除，剔除的具体标准改不了）。如果你的场景动态范围本来就小于15dB（比如近距离强LOS场景），这个阈值可能误删有效的弱径；如果动态范围很大（比如有遠距离强散射体），15dB可能不够激进，留下太多噪声。

## 3. UI 显示参数（不影响估计结果，只影响"怎么画"）

### 3.1 `relative_power`（绝对/相对功率开关）

刚改完：所有图表现在默认输出绝对dB（不做归一化）。这个开关现在功能完整，但**UI界面上完全没有暴露**——目前只能在生成JSON时由后端决定，前端用户没法在浏览器里切换"看绝对值"还是"看相对峰值归一化后的形状"。两种视图各有用处：绝对值看真实功率大小，相对归一化看波形细节对比（弱信号也能看清形状）。

### 3.2 `max_delay_bins` / `n_doppler_bins`

**这是什么**：纯粹是渲染分辨率——延迟轴/Doppler轴downsample到多少个点传给前端画图。不影响SAGE等算法本身的估计精度（算法用的是完整数据），只影响图表看起来够不够细。

**调大**：图更细，但JSON文件更大，前端渲染更慢（尤其是2D热力图，点数是两个轴的乘积）。
**调小**：文件小、渲染快，但可能错过两个采样点之间的窄峰（之前发现过这个问题：`frameStats.peakPowerDb`用完整分辨率算出28.38dB，但downsample后的`pdpCurve`只有23.04dB，因为真正的峰值bin被跳采样跳过了）。

### 3.3 colorbar 范围（`web/static/app.js::robustRange`）

刚改成真实min/max（之前是1~99分位裁剪）。如果某一帧/某个窗口出现异常高的功率值（比如直接耦合泄漏），会把整个colorbar的对比度拉得很低，其他正常范围的细节看不清——这是用真实范围的代价。如果以后觉得对比度不够用，可以考虑在UI上做成可切换（真实范围 vs 百分位裁剪），而不是写死成一种。

## 4. 现状：参数链路在哪里断掉了

```
真正起作用的地方（sage_adaptive.py 函数签名）
        │ 支持11个参数
        ▼
compute_adaptive_sage_tracks (ui_dataset.py)
        │ 只转发6个：window_size_frames, step_frames,
        │           delay_gate_distance_m, max_delay_bins,
        │           coverage_target, max_paths_hard
        ▼
build_measurement_dataset / analyze_one (pipeline/analyze.py)
        │ 一个都不转发，全部用 compute_adaptive_sage_tracks 的默认值
        ▼
AnalyzeRequest (frontend_app.py) / web/index.html 表单
        │ 只有 rxBinName, calBinName, carrierHz, txMode, txLat/Lon/Alt
        ▼
你在浏览器里能调的：0个 SAGE 参数
```

如果想让某个参数变成UI可调，至少要把它从断点处一路加回 `AnalyzeRequest` 和表单，缺一层都不行。
