## 四、Claude Code 的诞生故事

### 4.1 原型阶段

Boris 加入 Anthropic 后，花了一个月时间做各种"奇怪的原型"，大多数从未发布。他坚信：**要做好 AI 产品，必须理解"模型层之下的那一层"**，即模型本身的运行机制。

最初版本叫做 **Claude CLI**，纯终端界面，是他一个人开发的。内部演示后，只收到两个点赞。没有人相信一个编程工具可以是终端形态的。

### 4.2 灵感时刻

原型阶段的突破点：Boris 给模型一个 bash 工具，然后问它"我现在在听什么音乐？"

> 模型在没有任何额外指令的情况下，自己想明白了如何用 bash 工具来回答这个问题。

这个**涌现式工具使用**的瞬间，让他确信自己发现了某种真实的东西。

### 4.3 终端形态的选择逻辑

终端界面的选择，一半是现实原因（他是独立开发者），一半是战略判断：**终端是唯一能跟上底层模型迭代速度的界面**。当时，图形化界面的开发成本太高，无法跟上每隔几个月就大幅升级的模型能力。

### 4.4 增长拐点

- **2025年2月**：Claude Code 对外发布，初期并未爆火，终端界面对很多人来说过于陌生。
- **Opus 4 发布（Anthropic 首款 ASL-3 级别模型）**：增长真正进入指数曲线。
- **2025年11月**：Boris 个人跨越"100% AI 写代码"的门槛。
- **2026年初录制时**：增长曲线仍在加速——不是线性增长，是速度本身在加快。

## 五、"编程已解决"——这意味着什么？

### 5.1 精确定义

Boris 强调，他所说的"编程已解决"，并非指软件工程整体，而是特指：

> **将人类意图翻译成能运行的代码，这一具体动作，已经被 Claude 解决了。**

"思考架构、决定做什么、确保正确性"——这些仍然是未解决的问题。

### 5.2 他自己的工作方式变化

- 自 2025 年 11 月起，他 100% 的代码由 Claude Code 撰写
- 他仍然审查代码，保留人工检查节点
- 但实际的代码**书写**完全由 AI 完成
- 他依然是 Anthropic 最高产的工程师之一

### 5.3 Spotify 案例印证

录制当周，TechCrunch 报道：**Spotify 最优秀的工程师自 2025 年 12 月起就没有手写过一行代码**，全部依赖 AI。这不再是边缘现象，而是头部科技公司的主流实践。

## 六、接下来会发生什么：新的前沿

### 6.1 从"工具"到"同事"

Claude 已经开始：
- **主动查看 bug 报告和遥测数据**，提出自己的修复方案
- **建议接下来要开发什么功能**
- 作为"联合创始人级别的同事"参与决策，而不只是执行任务

### 6.2 "Builder"角色的崛起

Boris 认为，传统职能边界正在消融：

- **当前**：在 Claude Code 团队，每个人都编程——产品经理编程、工程经理编程、设计师编程、财务人员编程、数据科学家编程。
- **年底预测**："软件工程师"这个职位名称将开始消失，被"**Builder**（构建者）"替代——一个融合了产品思维、设计感知、技术编排和用户共情的复合角色。
- **或者**：所有人都是产品经理，所有人都会编程。

### 6.3 下一批被 AI 转型的职业

按照 Boris 的判断，紧随工程师之后被 AI 深度转型的职业是：

1. **产品经理**（AI 已能分析数据、提出需求）
2. **设计师**（AI 可生成界面，设计师转向决策和判断）
3. **数据科学家**（编程能力不再是门槛）
4. **非技术白领**（Cowork 的目标群体）

## 七、产品哲学：四大反直觉原则

### 原则一：故意"欠配资源"

> 与其给一个项目配五名工程师，不如只配一名——他会被迫用 Claude Code 自动化一切。

这不是节省成本，而是**通过约束逼出创新**。当团队"兵力不足"时，工程师必须依赖 AI，从而发现 AI 能力的边界，推动产品进化。

### 原则二：给工程师无限 Token

不要限制工程师使用 AI 的成本预算。**创新的想法来自于把 AI 用到极限的人**。Anthropic 有工程师每月消耗数十万美元的 Token。先找到有效的方案，再优化成本——不要反过来。

实用建议：**用最贵的模型，不要用便宜的模型省钱**。便宜的模型需要更多纠错，最终花的 Token 更多，效果也更差。

### 原则三：为六个月后的模型而构建

> 不要为今天的模型构建产品，要为六个月后的模型构建产品。

Claude Code 早期，AI 只写了 Boris 约 20% 的代码。但团队押注模型能力会指数级提升。当 Opus 4 和 Sonnet 4 发布时，产品立刻达到了产品-市场契合点。如果按照当时的模型能力设计产品，就会错过这个窗口。

### 原则四："苦涩的教训"——押注最通用的模型

Boris 引用 Rich Sutton 的经典博客文章《苦涩的教训》（The Bitter Lesson）作为团队核心原则：

> **更通用的模型，总会打败更专用的模型。**

在产品层面的含义：**不要在 AI 模型外面包裹复杂的工作流和编排脚手架**。不要把模型框死。给它工具，给它目标，让它自己找路径。那些精心设计的脚手架可能带来 10-20% 的性能提升，但下一代模型发布后，这些增益会被抹平——而没有脚手架束缚的产品会更快适配新能力。

## 八、"潜在需求"——Boris 最重要的产品原则

### 8.1 定义

**潜在需求（Latent Demand）**：观察用户以你从未预设的方式使用产品，那里就藏着下一个产品的雏形。

Boris 将其称为"产品开发中最重要的单一原则"，并将理论源头追溯到 Fiona Fung——Facebook Marketplace 的创始经理。

### 8.2 经典案例

- **Facebook Marketplace 的诞生**：Facebook Groups 中有 40% 的帖子是买卖交易。这是从未被设计的功能，却自发涌现，成为一个数十亿美元的产品。

### 8.3 Claude Code 的发现

Claude Code 发布后，Anthropic 观察到了大量非工程师用户的奇特使用方式：
- 有人用它种西红柿（获取种植指导）
- 有人用它分析自己的基因组
- 有人用它从损坏的硬盘中恢复婚礼照片
- 数据科学家自己安装 Node.js，通过终端运行 SQL 分析

这告诉团队：**Claude Code 的价值不只限于工程师**，非技术用户也有巨大需求。

### 8.4 AI 时代的新维度

Boris 提出了传统"潜在需求"理论的 AI 原生变体：**观察模型本身在尝试做什么，然后为之提供支持**。

> "产品就是模型。其他一切都是最小化的脚手架。"

不要预设工作流，让模型在给定工具和目标后自己找路径。Anthropic 内部称此为"on distribution"（在分布上）。

## 九、Cowork：10天构建，更大的野心

### 9.1 是什么

Cowork 是 Anthropic 为非技术用户打造的智能体工具，目标是自动化日常工作任务。

### 9.2 能做什么

- 缴纳停车罚款
- 取消订阅服务
- 管理项目电子表格
- 在 Slack 上给团队成员发消息
- 回复邮件
- 填写各类表单

### 9.3 构建过程

由一个小团队用 **Claude Code 在 10 天内** 完成构建，包括虚拟机和安全沙箱的搭建。

### 9.4 为什么比 Claude Code 更重要

> Claude Code 转型的是工程师。Cowork 转型的是所有人。

Cowork 上线即成为比 Claude Code 发布时更大的热门产品，且增长速度更快。其目标受众是全球数十亿白领，市场规模远超编程工具。

## 十、AI 安全：Anthropic 的三层防护

Boris 详细解释了 Anthropic 在 AI 安全上的体系：

### 第一层：机制可解释性（Mechanistic Interpretability）

由 Chris Olah 主导，在神经元级别研究模型内部结构：
- 理解概念是如何编码的
- 研究计划能力如何运作
- 在神经元层面检测欺骗行为
- 单个神经元可能通过"叠加"（superposition）现象对应数十个概念

### 第二层：评估（Evals）

在实验室环境中，将模型置于人工合成的场景中测试对齐程度。

### 第三层：真实世界监控（Real-world Monitoring）

以"研究预览"形式发布产品，研究智能体在真实环境中不可预测的行为。Claude Code 在对外发布前，在 Anthropic 内部使用了 4-5 个月，专门用于安全研究。

**重要价值观**：Anthropic 坚持安全工具开源，即便这些工具也适用于竞争对手的产品。他们发布了可兼容任意 AI 智能体的开源安全沙箱，不仅限于 Claude Code。

## 十一、印刷机类比：理解这场转型的历史坐标

Boris 最喜欢用的历史类比是**古腾堡印刷机**：

| 维度 | 15世纪印刷机 | 今天的 AI 编程工具 |
|------|------------|-----------------|
| 被解放的技能 | 阅读与写作 | 编写代码 |
| 原来的"特权阶级" | 抄书僧侣（识字率<1%） | 软件工程师 |
| 触发机制 | 古腾堡，约 1450 年 | Claude Code，2025 年 |
| 50年内产出变化 | 超过此前千年总和 | 代码量即将爆炸式增长 |
| 成本变化 | 下降 100 倍 | 类似趋势 |
| 长期识字率 | 从<1% 升至 70% | 编程将成为普遍技能 |
| 时代效应 | 引发文艺复兴 | 无法预知，但同等量级 |

Boris 还分享了一个令他印象深刻的细节：**他找到了一份 1400 年代的僧侣采访记录，那位僧侣对印刷机感到兴奋，因为它让他们可以从机械的抄书工作中解脱出来，专注于更有艺术性的工作——插图和装订**。Boris 在这个故事里看到了自己的写照。

## 十二、给开发者的实操建议

### 12.1 使用 Claude Code 的三个核心技巧

**技巧一：用最强的模型**
当前推荐 Opus 4.6，开启最大算力模式。便宜的模型看似省钱，但需要更多纠错，最终 Token 消耗更多。

**技巧二：善用计划模式（Plan Mode）**
在终端中按两次 Shift+Tab 进入计划模式，告诉模型"不要写代码"，先来回讨论方案。方案确定后再开启"自动接受编辑"，Opus 4.6 几乎能一次性做对。

**技巧三：探索不同界面**
Claude Code 运行于终端、桌面应用、iOS、Android、Web、Slack、GitHub、IDE 插件等多种界面，同一个 Agent 在所有这些平台上运行。找到适合你工作流的界面。

### 12.2 给 CTO 和技术负责人的建议

1. **故意给团队配少一点的人手**，让他们必须依赖 AI
2. **给工程师无限 Token 预算**，让他们在不考虑成本的情况下实验
3. **押注最通用的模型**，避免过度定制化的 AI 工作流
4. **为六个月后的模型能力设计产品**，而不是为今天设计
5. **观察"潜在需求"**——用户在用你的产品做你没想到的事情，那里藏着下一个产品

## 十三、关于编程技能的未来

### 短期（1-2年内）

手动编码技能将变得**无关紧要**，就像汇编语言——它仍然存在于底层，理解基础有一定帮助，但对绝大多数工程师来说不再必要。

### 长期

Boris 相信，就像印刷机让阅读从1%普及到70%，编程也会成为一种**普遍技能**，而不再是少数专家的特权。

### 对设计师的特别说明

Lenny 的调查显示，工程师和产品经理中有 70% 表示 AI 让他们更享受工作，但**设计师的数字只有 55%，且有 18% 表示更不享受**（是工程师比例的两倍）。

Boris 的解释是：Anthropic 的设计师因为能自己写代码而解放了自己。但评论者指出，这回避了核心问题：**"能解放自己"不等于"AI 能做设计"**。视觉思维、空间推理、用户体验设计——这些 AI 还没有真正触及。

## 十六、总结与启示

### 核心叙事

这不是一个关于某个工具的故事，而是关于**一种范式的消亡和另一种范式的诞生**。正如古腾堡印刷机终结了抄书僧侣的时代，Claude Code 正在终结"手工编写代码"的时代。

### 对不同群体的启示

**对工程师：**
- 不要抗拒，要拥抱。Boris 从"偶尔用 AI"到"100% 依赖 AI"的时间不到一年。
- 把时间从"写代码"转移到"想清楚做什么"。
- 学会提示词工程和 AI 编排，这是新的核心技能。

**对产品经理：**
- 你们的角色正在扩大而不是收缩——AI 增强了你们的执行能力。
- "决定做什么"仍然是人类最重要的工作。

**对设计师：**
- 这是最不确定的领域——视觉思维是 AI 尚未触达的地方。
- 但"能自己实现想法"的设计师将更有价值。

**对领导者和管理者：**
- 用"故意欠配资源 + 无限 Token"的策略逼出创新。
- 押注通用模型，不要过度定制化。
- 观察"潜在需求"，这是寻找下一个产品机会的最可靠方法。

**对所有人：**
- 这场转型将"非常具有破坏性，对很多人来说非常痛苦"——这是 Boris 自己说的。
- Anthropic 已经雇用了经济学家、政策专家和社会影响专家来思考这些问题。
- 社会层面的对话不能再等待了。

### 最后一个问题

Boris 预计今年年底，每个人都将既是产品经理，又会编程。软件工程师的头衔将开始消失。这是预言还是愿景？从他的数据来看，这条曲线已经不需要任何信仰就能看见。
